谈一谈Docker应用
2019-02-26 00:25:22 约3.9k字 约14分钟 # Docker

楔子

第一次听说Docker这个名词是2014年左右，当时研究并不深入，而且在非互联网公司里面对新技术的引入和应用往往是滞后的（并不是不创新，而是处于安全及稳定性的考虑）。当时只是知道Docker最简单的说明是应用及虚拟化，国内的应用更是少之又少，镜像仓库也远没有现在这样成熟。

不过我想推广缓慢的主要原因可能在于其他两个原因：

  1. 当年企业级应用变更主流的讨论是SOA（面向服务的架构），实际场景中大家都在研究Dubbo，微服务的概念虽然已经有了，拆分边界和应用案例并不清晰
  2. Spring全家桶的入口Spring Boot刚刚面世，更不用说Spring Cloud了

时过进迁，重新捡起技术之后发现Spring Boot 2.X已经强大到一逼，而Spring的官网已经非常明确而自豪的对外宣布，Java web工程未来的开发趋势，就是下面这张图Build Anything | Coordinate Anything | Connect Everything：

未来已来，代码构建器除了简单的增删改查以外，通过模版引擎的应用，已经可以做到表单的配置和自动生成，机器人自动完成架构搭建和业务实现的时代还远吗？也许未来80%的开发人员都要下岗，谁知道呢？

有点扯远了，说回Docker，我想理论知识尽量少说，百度上到处都是，而且还有各式各样的视频教程，这里单纯说一说Docker的应用以及最最最常用的命令。

Docker是什么

这一段是从Docker官网抄的：

Docker 使用 Google 公司推出的 Go 语言 进行开发实现，基于 Linux 内核的 cgroup，namespace，以及 AUFS 类的 Union FS 等技术，对进程进行封装隔离，属于 操作系统层面的虚拟化技术。由于隔离的进程独立于宿主和其它的隔离的进程，因此也称其为容器。最初实现是基于 LXC，从 0.7 版本以后开始去除 LXC，转而使用自行开发的 libcontainer，从 1.11 开始，则进一步演进为使用 runC 和 containerd。

Docker 在容器的基础上，进行了进一步的封装，从文件系统、网络互联到进程隔离等等，极大的简化了容器的创建和维护。使得 Docker 技术比虚拟机技术更为轻便、快捷。

太复杂看不懂是不是？简单举例子说明，以往软件公司发布产品到客户方进行部署上线是什么样的流程？

  1. 开发人员完成开发，打包提交测试
  2. 测试人员搭测试环境进行测试，如果出现Bug则返回开发
  3. 开发人员进行Bug验证，如果能够重现则修改Bug并重新提交测试
  4. 开发人员如果无法重现则需要跟测试人员沟通，确认排查环境因素（一般这个周期会很恐怖）
  5. 测试完毕后，开发人员进行打包发版，提交给现场实施人员
  6. 实施人员拿到包进行UAT环境部署，部署后提交用户测试，如果出现Bug则返回给实施人员
  7. 实施人员需要检查：用户操作、部署流程、UAT环境等因素，如果如不能解决，则返回给开发人员进行验证
  8. 开发人员进行验证，如果能够重现则修改Bug，并重复步骤4、5、6
  9. 实施人员部署到生产环境，并进行最终测试，如果出现Bug则重复步骤7、8

可以看出，在整个软件上线的生命周期中，环境部署和环境不一致会造成很大的问题，Docker就完美的解决了这个问题，如果整个流程使用Docker的话，那么流程会大大简化，比如：

  1. 开发人员完成开发，打成Docker镜像提交测试
  2. 测试人员启动镜像进行测试，如果出现Bug则返回开发
  3. 开发人员进行Bug验证，如果能够重现则修改Bug并重新提交测试（有了Docker之后，如果开发人员按照测试步骤进行操作，99%可以重现问题）
  4. 测试完毕后，开发人员打成Docker镜像提交给现场实施人员
  5. 实施人员启动镜像，用户进行UAT测试，如果出现Bug则返回给实施人员
  6. 实施人员需要检查：用户操作（有了Docker之后，99%的问题是用户操作不当，或者是测试人员的测试用例覆盖不足），如果如不能解决，则返回给开发人员进行验证
  7. 开发人员进行验证，如果能够重现则修改Bug，并重复步骤4、5、6
  8. 实施人员启动镜像，并进行最终测试，如果出现Bug则重复步骤7、8

Docker对外的口号就是：一次打包，到处运行，且运行环境完全一致。当你完成开发后，你把Tomcat、Java环境和web应用打包成一个镜像进行封装，那么这个镜像可以到处复制，只要装有Docker的机器都可以提取镜像并直接运行和你打包的环境相同的一个应用，数据库、中间件、Web应用这些统统可以打成镜像。

更为重要的一点，和传统虚拟机相比：

  1. Docker的启动是秒级的
  2. Docker的磁盘使用是MB
  3. Docker的性能接近原生
  4. Docker的单机支持上千个容器

安装Docker

在Mac OS下安装Docker

在Mac OS下安装Docker异常简单，在官网注册后直接点击Get Docker下载安装文件即可。不过因为一些众所周知的原因，要想优雅的抽取镜像，还需要配置Registry mirrors

https://registry.docker-cn.com

在CentOS下安装Docker

在CentOS下安装Docker同样简单，不过需要先判断内核版本，Docker 要求 CentOS 系统的内核版本高于 3.10

  1. 首先检查内核版本

​ uname -r

  1. 下载并安装，这里需要注意，在2017年3月之后，docker 在原来的基础上分为两个分支版本: Docker CE 和 Docker EE。Docker CE 即社区免费版，Docker EE 即企业版，强调安全，但需付费使用。

    安装一些必要的系统工具， yum-util 提供yum-config-manager功能，另外两个是devicemapper驱动依赖的：

    sudo yum install -y yum-utils device-mapper-persistent-data lvm2

    因为一些总所周知的原因，需要设置国内的yum源:

    sudo yum-config-manager --add-repo http://mirrors.aliyun.com/docker-ce/linux/centos/docker-ce.repo

    更新 yum 缓存：

    sudo yum makecache fast

    安装Docker：

    sudo yum -y install docker-ce

    启动 Docker 服务:

    sudo systemctl start docker

Docker的使用

如果只是正常使用的话，需要了解的命令非常有限，最常用的就是下面五个

列出镜像列表

使用 docker images 来列出本地主机上的镜像

查找镜像

使用 docker search [镜像名] 命令来搜索镜像，比如搜索我打包的镜像

搜索的时候可以直接搜索镜像名，也可以搜索作者名/镜像名，他会按STARS的数量进行排序，OFFICIAL说明是官方的镜像

获取一个新的镜像

使用 docker pull [镜像名] 命令来获取镜像

如果不带版本号，就会自动下载最新版本的镜像，或者如果你特别明确的知道镜像的名称，可以直接使用 docker run [镜像名] 本次仓库没有这个镜像就会自动去pull

删除镜像

使用 docker image rm [IMAGE ID] 命令来删除镜像，比如删除上面获取的镜像

运行镜像

使用 docker run [镜像名] 命令来启动镜像，这里不推荐直接使用，因为一般来说启动时需要配置宿主机和镜像之间的端口映射、磁盘映射，建议用Dockerfile来运行，放到Docker构建里面一起说

查看镜像运行状态

使用 docker stats 命令来查看镜像运行状态

我这台云服务器是2C4G的，可以看出容器的运行非常精细化，毕竟穷啊！

Docker的构建

Dockerfile

还是一句话，Dockerfile可以把它理解成一个Windows上的BAT文件（但实际上还是有区别的），可以看成是一组顺序执行的dockerfile配置语句，不过既然说Docker构建，还需要简单说一下Docker的构成，理论知识不说了，这样来理解，Docker的构成就像洋葱一样，一层包一层，核心是原始容器，它可以是空的，也可以是基本的Java环境，就像下面的图一样

这样就引出了Dockerfile的第一个命令

FROM，指定基础镜像，比如可以写成FROM java:8 ，这就是指定了基础镜像为java，且tag为8的版本，对应jdk为1.8

EXPOSE，指定镜像端口号，比如可以写成EXPOSE 8080，说明这个镜像对外的端口为8080

VOLUME，指定数据持久化的文件夹，这里应该高亮来说，docker镜像运行是像redis一样的，一旦镜像停止，里面发生的任何数据变化是不会保存的！！！因此需要指定一个文件夹，这个文件夹映射到宿主机上去，所有需要保存的数据都应该放在这个文件夹里面！！！这里可以写成VOLUME /tmp，因为Spring Boot使用的内嵌Tomcat容器的默认工作目录为/tmp

ADD，把指定路径的内容放到镜像的指定路径中，这个主要用于把打包的web文件扔到镜像的web容器里面，比如这句话ADD jmall-admin.jar /app.jar它会把我打包好的Spring Boot文件扔到容器的根目录里面并且重命名为app.jar

RUN，这个很容易理解，就是执行docker命令，对应上面的docker run

ENTRYPOINT，指定镜像的默认入口，比如ENTRYPOINT ["java","-jar","/app.jar"]，这个特别眼熟吧java -jar /app.jar

使用Dockerfile

使用Dockerfile非常简单，最省事的方法就是直接进入到Dockerfile同一层级的目录中直接运行命令docker build，创建完毕的镜像会直接放到你的本地仓库，可以使用docker images命令来查看

提交你的镜像

做好了镜像可以直接把镜像文件提交到私服或者docker hub上面去给其他人分享了，提交镜像也很简单，我是直接用IDEA进行的提交，如果在服务器上的话也可以用命令

首先登陆docker hub

docker login

回车后会让你输入用户名和密码

看到Login Succeeded的时候就可以进行提交了，docker push wellmoonloft/jmall这里需要注意，前面你的名字，要跟你登陆的用户名一致，并且，由于众所周知的原因…提交过程非常漫长，所以还是强烈的建议用工具提交吧，至少还有个进度条可以看。

把镜像保存成文件复制给别人

有时候我们打包的镜像属于内部系统，不方便push到公网上，或者我们公司没有私服的情况下，要把镜像交给别人就需要保存成一个tar文件复制给别人，镜像保存也很简单，执行命令docker save -o [要保存成的文件名] [要保存的镜像文件或者IMAGE ID]，完成后会在你执行命令的路径下有一个tar包，这个包可以直接复制给别人用，需要注意的是如果是Cent OS或者Mac OS，注意路径的授权

把镜像导入本地仓库

收到别人给你的镜像tar文件不能直接使用，需要把它导入你本地的仓库，也很简单，执行命令docker load < [文件名]，对应上面的文件就应该这样写docker load < jmall.tar

Docker Compose

如果只是运行一个简单的单体应用，到这里就足够了，但是在生产中往往是多个应用一起合作完成一个系统的搭建，如果再加上负载均衡，可能要同时启动或者停止多个镜像，并且统一管理，在Docker里面有一个很好用的机制叫做Docker Compose来解决这个问题，这里需要引入两个熟悉的名词。

  • 服务 (service)：单一的容器镜像，可以是一个数据库、一个Web应用
  • 项目 (project)：由一组关联的应用容器（service）组成的一个完整业务单元

不说概念，看实例：

version: '2'这个不是随便乱写的，这个版本是主机上安装的Docker Compose版本

services是固定写法，说明下面的都是一个个的镜像

jmall-adminservice的名字，对应下面image里面的镜像名

ports，在构建镜像的时候指定了镜像端口为8080，但是实际上直接run镜像的时候是无法访问的，因为并没有把镜像抛出的端口跟宿主机端口进行映射，这里就是把宿主机的8080跟镜像中的8080进行映射，当然如果不用Docker Compose也可以直接docker run -p 8080:8080进行映射

environment是强制变更Spring Boot配置文件的内容，这里我写的是指定开发环境

使用Docker Compose

使用Dokcer Compose也很简单，进入Dokcer Compose同级路径，使用命令docker-compose up即可

思考

技术进步真心是无止境的，而且是相互关联非常紧密的，微服务如果没有Spring Boot、没有Docker、没有Dubbo、没有Zookeeper、没有Redis这些技术，想要全面推广真的很难，通过传统的产品+服务的形式很难存活，市场逼迫技术进行开源，中国开源界也越来越成熟了，开源不等于免费，而是相互学习、相互扶持，共同寻找落地案例、最佳实践、这才是良性的循环。

技术的未来一定不是封闭的，而是技术组合+封装服务的方式。

Gitee的出现，现在越来越多优秀的国产开源产品，如JFinal、xxl-sso、jeesite这些或是独立开发、或是小团队开发的产品越来越多了。

而大厂也在拥抱开源，不用说贡献各种国服、推出Dubbo等各式各样开源产品的国内大神、业界良心的阿里，各种被吐槽政策推动技术，但至少还有一个echarts还能看的出是互联网公司的百度，再勉强搭上一个对各式各样的收费方式研究到头发丝的腾讯。

能赶上这个时代，真心太美好了。
